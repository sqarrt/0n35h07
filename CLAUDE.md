# OneShot - аркадный шутер от первого лица

## Development Rules
- При разработке ВСЕГДА пользуйся плагином Superpowers
- После каждой правки прогоняй тесты в headless режиме, актуализируй тесты в соответствии с правками
- Для запуска в headless режиме пользуйся npm run test
- Когда выполняешь что-то из TODO.md ВСЕГДА зачеркивай выполненное
- НИКОГДА не делай коммит самостоятельно. Только по требованию пользователя

1. Одна ответственность с первого файла
Когда пишешь новую систему (оружие, физика, AI) — сразу выносить в отдельный хук/файл, не копить в Game.tsx. Признак что пора разбивать: файл > 150 строк или содержит более 2-3 несвязанных useRef/useEffect.

2. Константы отдельно с первого числа
Магические числа в коде (1500, 0.4, 400) — сразу в constants.ts. Когда их нет, невозможно понять что регулирует что, и они расползаются по файлам.

3. Утилиты на первом дубликате
Первый раз скопировал блок кода (raycast, анимация луча) — это сигнал сделать утилиту/компонент. Второй дубликат — уже технический долг.

4. Архитектурный слой для пропов
Когда App → Game передаёт 5+ callback-пропов — это признак пора ввести useReducer/context. В этом проекте это момент когда добавился 4-й flash-эффект.

5. Тесты на уровне систем, не только E2E
Playwright-тесты не поймали ни одной проблемы во время рефакторинга. Нужны unit-тесты хуков (useBeamWeapon, useShieldSystem) — они дешевле и точнее показывают что сломалось.

6. userData.noRaycast как соглашение от начала
Любой меш который не должен быть raycast-целью — маркируется сразу при создании. Не потом.

7. Короткие итерации с проверкой
После каждого нового геймплейного элемента — прогон тестов + TypeScript. Не копить правки.

8. После правок делай ВДУМЧИВОЕ ревью своего кода
Чтобы не допустить ситуаций, когда нужен массовый рефакторинг.
