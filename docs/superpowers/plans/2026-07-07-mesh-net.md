# mesh-net Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Децентрализация: полный меш «у каждого факта ровно один владелец» — каждый пир симулирует своих игроков (себя + свои боты), судит смерти своих, выводит счёт/стрики из канонического потока `kill`; хост-авторитет, лаг-комп и предикшн-реплей удаляются.

**Architecture:** Роли `host|client` в Match/NetSession заменяются одной симметричной `peer`. Владение: `owners: Map<playerId, PeerId>` едет из комнаты (создатель владеет ботами). Выстрел: локальный raycast (уже есть) → `fired` broadcast + адресный claim владельцу жертвы → жертва судит по СВОЕМУ состоянию («щит побеждает») → broadcast `kill`/`block` → каждый пир применяет одинаково (деривация счёта/стриков/SINGULARITY локальна). Снапшоты: каждый пир шлёт состояние своих игроков (30 Гц); интерполяция уже receive-time per-player — N отправителей работают без общего клока. Лобби-слой (RoomSession, роли host/client в КОМНАТЕ) не меняется.

**Tech Stack:** TypeScript 6 strict, Vitest + `createLoopbackHub` (N пиров), Playwright (`?net=bc` шина).

**Спека:** `docs/superpowers/specs/2026-07-07-multiplayer-4p-mesh-design.md` §5–6, ветка 3 §11.

- Каждая задача заканчивается компилируемым (tsc) и зелёным (затронутые юниты) состоянием. Переходная стратегия: новые симметричные пути добавляются РЯДОМ (задачи 1–4), катовер одним махом (задача 5), удаления после катовера (задача 6).

### Task 5: Катовер — симметричные Match('peer') и NetSession

Самая объёмная задача: роль `peer` включается для ВСЕХ матчей, старые ветвления host/client в update/applyPhysics/NetSession сливаются в один путь по владению.

**Ключевые контракты:**
- **Match.update (один путь):** controllers.update (только owned имеют контроллеры) → players.update → owned-стрелки: fired-outcome → emit `fired` + claim-очередь/judgeClaim (Task 4) → applyPhysics: owned → KCC (`computeColliderMovement`), не-owned → `setNextKinematicTranslation(nextRemoteTranslation())` (интерп, как клиент сегодня) → combat/respawn/HUD только для owned жертв (respawn owned: по таймеру владельца + emit `respawn`).
- **serializeSnapshot:** только owned-игроки; `Snapshot { tick, players }` (ackTick/buffered/restore — vestigial, зануляются, удаление полей в Task 6).
- **applyPeerSnapshot(from, snap):** применять ТОЛЬКО к игрокам с `ownerOf(id) === from` (атрибуция — защита от рассинхрона), `p.applyNetState(...)` как сейчас.
- **applyPeerEvent(from, e):** `kill`/`block` — принимать только если `ownerOf(e.victim) === from` (жертва — единственный судья); `fired`/`move`/`respawn` — только `ownerOf(e.id) === from`; `ready` — только `ownerOf(e.id) === from`.
- **Фазы:** каждый пир: локальный ready → `markReady(myIds)` + emit `{t:'ready', id}`; ВСЕ пиры ведут readySet; только СОЗДАТЕЛЬ (владелец слота 0) при полном комплекте переключает `phase='countdown'` и шлёт `phase`-сообщение (существующий PhaseMsg); остальные ставят локальный `countdownEndsAt` по приёму (skew ≤ RTT — осознанно, часы машин не синхронны; это и есть «штамп создателя»). `tickPhase` countdown→live локален. Таймер матча: `matchEndsAt` от live локально (skew ≤ RTT); событие `time` не шлётся (каждый тикает своё), `matchEnd` не шлётся (endMatch по своему таймеру / left-правилу).
- **NetSession (симметричный):**
```ts
// every peer, no role branching:
net.on('snapshot', (p, from) => match.applyPeerSnapshot(from, p as Snapshot))
net.on('event',    (p, from) => match.applyPeerEvent(from, p as MatchEvent))
net.on('hit',      (p, from) => match.judgeIncomingClaim(from, p as HitClaim))  // gate: I own claim.hitId
net.on('phase',    (p, from) => match.applyPhase(p as PhaseMsg, from))          // gate: from === creator peer
net.onPeerLeave(peer => match.handlePeerLeft(peer))                             // ALL players owned by peer → left
afterUpdate(now):
  for (const e of match.drainEvents()) net.broadcast('event', e)
  for (const { to, claim } of match.drainClaims()) net.send(to, 'hit', claim)
  if (match.phaseDirty() && match.iAmCreator) { net.broadcast('phase', match.serializePhase()); match.clearPhaseDirty() }
  if (throttle30Hz) net.broadcast('snapshot', match.serializeSnapshot())
```
- **`handlePeerLeft(peer)`:** все id с `ownerOf(id)===peer` → `handlePlayerLeft(id)` (боты уходят с владельцем — спека; правило «<2 команд → конец» уже есть). Клиентская ветка «ушёл = хост id 0» УДАЛЯЕТСЯ (в меше уход создателя ≠ конец матча).
- **Game.tsx:** `requestReady` → `match.requestLocalReady()` (ready своих + emit); `clockNudge` вызов удаляется (driver.advance(dt, 0)); demo-гейт `role !== 'host'` снимается; `emit`-гейт (Match.ts:945 host-only) снимается — каждый пир эмитит события СВОИХ фактов.
- **App.tsx:** `matchRole = 'peer'` всегда; RoomSession role остаётся лобби-понятием.

**Steps:**
- [ ] Переписать NetSession.test на `createLoopbackHub(['A','B','C'])` + MatchNet.test на 3 симметричных Match (owners по пирам): снапшот A применяется у B/C только к игрокам A; kill от владельца жертвы разлетается и счёт сходится на всех трёх; claim B→A судится у A; фазовый ритуал (ready от всех → создатель штампует → у всех countdown); уход пира C → его игроки left у A и B, матч жив. ### Task 7: Юниты протокола смерти на LoopbackHub (основное покрытие спеки §10)

**Files:** Create `tests/unit/mesh.deathProtocol.test.ts`

Харнес: 3 полных стека (Match+NetSession) на `createLoopbackHub(['A','B','C'])`, owners: A владеет 0 (+бот 3 в 2v2-кейсах), B — 1, C — 2; `forceLiveForTest`; прокачка `afterUpdate` вручную.

- [ ] Кейсы (падают до/зелёные после доводки):
  1. **Сходимость счёта**: B стреляет-попадает в 2 (C-жертва) → у A/B/C kills/deaths идентичны.
  2. **Двойной килл**: claim'ы от A и B в жертву C в одном флаше → ровно один kill-event, счёт一致, второй стрелок без фрага.
  3. **Щит побеждает**: у жертвы C локально активен щит в момент прихода claim → block у всех, предсказание стрелка откатилось (`reviveFromFalsePrediction`).
  4. **Потерянный claim**: claim не доставлен (не слать) → grace-таймаут стрелка ревайвит жертву из её снапшотов.
  5. **Teammate**: 2v2, claim от тиммейта → игнор без событий.
  6. **Уход пира**: C уходит → его игроки left у A/B; при остатке ≥2 команд матч жив; боты C ушли вместе с ним.
  7. **Фазовый ритуал**: ready каждого доходит до всех; countdown стартует только по штампу создателя A.
- [ ] Прогон → PASS → Commit `test(mesh): протокол смерти на LoopbackHub — сходимость, гонки, щит, уход`.

## Заметки исполнителю

- **Порядок update() пира** (инвариант из CLAUDE.md сохраняется): syncFromBody → controllers.update (owned) → players.update → owned-fire→claims/judge → applyPhysics (owned KCC / remote interp) → owned combat-derivation/respawn/HUD → lateUpdate.
- **`emit`** становится «положить в pendingEvents + применить локально»: владелец факта применяет свой факт немедленно (не ждёт эха — транспорты самодоставку не делают), остальные — по приёму. Следить за двойным применением: `applyKill` вызывается у владельца в `judgeClaim`, у остальных в `applyPeerEvent` — один раз у каждого.
- **Нокбэк** (`maybeKnockback`): только для owned-игроков, против живых (интерполированных) позиций остальных; перемотка vt удаляется.
- **Демо**: гейт снимается; `DemoRecorder` пишет точку зрения пира (спека §12) — ничего в классе не менять.
- **`__debugRole`** и прочие дебаг-глобалы: проверить потребителей в e2e; `__debugRole` в меше может возвращать 'peer' + `iAmCreator`.
