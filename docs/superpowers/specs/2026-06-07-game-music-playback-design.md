# Музыка в матче: воспроизведение слоёных стемов — дизайн

**Дата:** 2026-06-07
**Ветка:** `feat/audio`
**Статус:** утверждён к реализации

## Цель

Проигрывать в матче процедурную фоновую музыку, собранную из готовых
рендер-стемов (`render_opus/`). Это «подпроект 2» из плана музыки: игра играет
**только готовые рендеры** (никакого Strudel/AGPL внутри игры), детерминированно
от сида матча, **одинаково у обоих игроков**.

## Исходные данные (факты по ассетам)

- **59 стемов** в 4 ролях: `bass / kicks / lead / sfx`.
- Все файлы **строго одинаковые**: длительность `8.0065 с`, `48 кГц`, стерео, Opus.
- Это лупы одной длины и темпа — созданы, чтобы их **слоить и зацикливать** вместе.
- `8.0065` ≠ `8.000`: лишние `~0.0065 с` (≈312 семплов @48к) — паддинг кодека Opus
  (pre-skip). **Музыкальная длина лупа = ровно 8.000 с.** Зацикливать надо по ней,
  не по длине файла, иначе слои расфазируются и подмена не попадёт в долю.

## Поведение (что музыка делает в матче)

Модель **C (эволюционирующая/генеративная)** с фиксированным интро:

- Матч начинается с **KICKS + BASS**.
- Позже вступают остальные роли (`lead`, `sfx`).
- В течение матча директор подменяет часть стемов из библиотеки — трек мутирует,
  оставаясь в темпе/строе (все стемы и так совместимы — один контракт).
- **Конкретные правила подмены тюнятся после плейтестов.** Сейчас закладываем
  только интро (kicks+bass → позже остальное) + механику подмены на границе лупа.

### Детерминизм / синхрон

- Выбор стемов и моменты подмены — **чистая функция от `(seed, loopIndex)`**.
- `seed` выводится из **лобби-кода** (общий у хоста и клиента, уникален на матч) —
  ноль нового сетевого трафика.
- «Идентично у обоих» означает **одинаковую последовательность аранжировок**
  (те же стемы на тех же `loopIndex`), а **не** семпл-в-семпл синхрон между
  машинами (он недостижим по WebRTC и не нужен — игроки в разных комнатах).

## Архитектура

Новый слой `src/game/audio/`. Web Audio — браузерная зависимость, поэтому
движок воспроизведения относится к «хост-слою» (как рендер), а правила
композиции — чистый TS (юнит-тестируемо). DIP между ними.

### Юниты

**1. Библиотека стемов — `src/game/audio/stems.ts`**
- Ассеты переезжают: `render_opus/<role>/*.opus` → `src/assets/music/<role>/*.opus`.
- Манифест собирается через `import.meta.glob('../../assets/music/*/*.opus')`;
  **роль = имя папки** (как в музыкальной памяти проекта). Vite сам хеширует и
  раздаёт файлы.
- Экспортирует `StemLibrary`: `Record<Role, StemRef[]>`, где `StemRef = { id, url }`.

**2. `MusicDirector` — `src/game/audio/MusicDirector.ts` (чистый TS, без Web Audio)**
- **Единственное место музыкальных правил.** Именно его крутят при тюнинге —
  хоть константами, хоть переписав алгоритм.
- `compose(seed: number, loopIndex: number, library: StemLibrary): Arrangement`
  где `Arrangement = VoiceSpec[]`, `VoiceSpec = { role, stemId, gain }`.
- Детерминированный PRNG, посеянный от `hash(code)` (+ `loopIndex` для эволюции).
- Правило интро: на ранних `loopIndex` в аранжировке только `kicks`+`bass`; с
  некоторого `loopIndex` добавляются `lead`/`sfx`. Пороги/набор — локальные
  именованные константы в этом файле.
- Чистая и детерминированная: одинаковые `(seed, loopIndex)` → одинаковый
  `Arrangement`. Полностью юнит-тестируема без браузера.

**3. `IMusicEngine` (DIP) + `WebAudioMusicEngine` — `src/game/audio/`**
- Интерфейс: `load(library)`, `start(startTimeSec)`, `applyArrangement(arr, atTimeSec)`,
  `stop()`, `dispose()`, `setMasterGain(g)`.
- `WebAudioMusicEngine`:
  - Владеет одним `AudioContext` + мастер-`GainNode`.
  - Декодит каждый Opus → `AudioBuffer` один раз, кэширует по `stemId`.
  - **Lookahead-планировщик** (паттерн «A Tale of Two Clocks»): таймер с упреждением
    смотрит на `ctx.currentTime`, и для каждой границы лупа планирует голоса как
    разовые `AudioBufferSourceNode`, стартующие точно на `boundaryTime`.
  - Луп по **`LOOP_SECONDS = 8.0`** (музыкальная длина), не по `buffer.duration`.
  - Каждый голос идёт через свой `GainNode`; вход/уход голосов — кроссфейд
    `linearRampToValueAtTime` на границе (без щелчков).
  - На каждой границе зовёт коллбэк «дай аранжировку для loopIndex N» (его даёт
    `MatchMusic` через директора) и диффит против текущей: общие голоса тянутся,
    новые — фейд-ин, исчезнувшие — фейд-аут.
- Фейк-`IMusicEngine` для юнит-тестов (записывает вызовы, без реального аудио).

**4. `MatchMusic` (связка) — `src/game/audio/MatchMusic.ts`**
- Конструируется с `(seed, library, engine, director)`.
- `loopIndex = floor((now − liveStartSec) / LOOP_SECONDS)`.
- Подписывается на границы лупа движка → `director.compose(seed, loopIndex, library)`
  → `engine.applyArrangement(...)`.
- Жизненный цикл завязан на фазу матча: старт на `live`, стоп на конце матча /
  `dispose`.

### Интеграция с `Match` / `Game`

- `code` лобби прокидывается: `Game props → MatchOptions → Match`. `Match` выводит
  `seed = hash(code)` и создаёт `MatchMusic`.
- Музыка чисто **локальная** (не симулируется, не гоняется по сети) — но управляется
  общим сидом, поэтому одинакова у обоих. Никаких новых сетевых сообщений.
- **Autoplay:** `AudioContext` стартует заблокированным; вход в матч идёт через
  клики (меню → лобби → «ГОТОВ»), поэтому `ctx.resume()` зовём на жесте готовности.
- Старт музыки — на переходе в `live` (после отсчёта).

### Константы (примерные, локальные; точные значения — при реализации)

- `LOOP_SECONDS = 8.0` — музыкальная длина лупа.
- `SCHEDULE_LOOKAHEAD_SEC`, `SCHEDULER_TICK_MS` — параметры lookahead-планировщика.
- `CROSSFADE_SEC` — длительность кроссфейда входа/ухода голоса.
- `INTRO_LOOPS_KICK_BASS` — сколько лупов играет только kicks+bass до вступления
  остального.
- `MASTER_GAIN_DEFAULT` — стартовая громкость музыки.

## Что НЕ делаем сейчас (YAGNI)

- Адаптацию под геймплей (интенсивность перестрелок) — правила подмены пока
  тривиальные, расширим после плейтестов.
- Дебаг-панель/внешний JSON-конфиг тюнинга — правим код `MusicDirector` напрямую.
- Настройку громкости в UI-настройках — мастер-гейн пока константа (добавим позже).
- `prebake.strudel` в игру не тащим (Strudel-прелюдия редактора, AGPL-сосед, для
  воспроизведения рендеров не нужна) — добавляем в `.gitignore`.

## Тестирование

Следуя стратегии проекта (Rapier/WebAudio/decode не идут в jsdom):

- **Юнит (`tests/unit/`)**: `MusicDirector.compose` —
  - детерминизм: одинаковые `(seed, loopIndex)` → одинаковый `Arrangement`;
  - интро: ранние лупы содержат только `kicks`+`bass`;
  - вступление: с порога появляются `lead`/`sfx`;
  - валидность: все `stemId` существуют в переданной библиотеке.
  - `MatchMusic` с фейк-движком: правильный `loopIndex` от времени, вызовы
    `applyArrangement` на границах.
- **e2e (`tests/*.spec.ts`)**: дымовой — вход в матч поднимает `AudioContext`
  (state `running`) и планирует ≥1 источник; проверяем через дебаг-глобал
  `__debugMusic` (текущий `loopIndex` + активные `stemId`).
