# OneShot — игровые SFX: дизайн

Дата: 2026-06-08 · Ветка: `feat/audio` · Статус: согласовано, готово к плану

## Проблема и цель

Соседний редактор прислал 18 готовых звуковых эффектов (см. `GAME_SFX_DELIVERY.md`): выстрел, щит,
прыжок/рывок/приземление, смерть/респавн, отсчёт, UI-клики и два лупа. Нужно подключить их в игру:
3D-позиционно для геймплея (звуки игроков идут из их позиции, соперника слышно «откуда»), 2D для UI и
отсчёта. Музыка матча — отдельная подсистема, её не трогаем.

## Формат и файлы

- Источник: `oneshot_music_editor/render_opus/game_sfx/<event>.opus` (48 кГц, libopus). Копируем в
  **`src/assets/sfx/<event>.opus`**. Имя файла = id события.
- Манифест **`SFX_LIBRARY: Record<SfxEvent, string>`** через `import.meta.glob('../../assets/sfx/*.opus',
  { eager:true, query:'?url', import:'default' })` (как `STEM_LIBRARY` у музыки).
- `ghost_loop` копируем, но **не подключаем** — в игре нет состояния «ghost form». Резерв на будущее.
- Звуки НЕ нормализованы по громкости (так в доке) → задаём `SFX_GAIN` на стороне игры.

## Движок (DIP, как у музыки)

Один экземпляр на всё приложение (живёт на уровне `App`), чтобы меню/лобби тоже играли 2D-звуки.

- **`ISfxEngine`** (граница абстракции):
  - `load(): Promise<void>` — декод всех буферов.
  - `attach(camera, parent): void` — на входе в матч: вешает listener на камеру, задаёт `parent`
    (`match.root`) для позиционных нод. `detach()` — на выходе.
  - `playAt(event, pos: Vec3, gain?): void` — позиционный one-shot.
  - `play2D(event, gain?): void` — непозиционный one-shot (UI/отсчёт).
  - `startLoop(event, key, target: Object3D): void` / `stopLoop(key): void` — лупы на меше игрока.
  - `setMasterGain(g)`, `dispose()`.
- **`ThreeSfxEngine`** — реализация на `THREE.AudioListener` (создаётся в конструкторе → один
  AudioContext; `setMasterGain` = `listener.setMasterVolume`). Позиционные звуки — `THREE.PositionalAudio`
  (one-shot: создаётся, ставится на позицию/в `parent`, играет, удаляется по `onEnded`; луп:
  `setLoop(true)`, привязан к `target`, живёт до `stopLoop`). 2D — `THREE.Audio`. Декод через
  `decodeAudioData` (срезает Opus pre-skip → атака на первом семпле). Контекст резюмится по первому
  жесту (клик в меню).
- **`FakeSfxEngine`** — для юнитов (three-аудио/WebGL в jsdom не идёт; тестируем логику триггеров).
- Спатиальные параметры (`refDistance`, `rolloffFactor`, `maxDistance`) — константы движка под
  компактную арену; тюнятся на слух.

## Проводка

- **`App`** создаёт `sfxEngine` один раз, кладёт в React-контекст (`SfxProvider` + `useSfx()`),
  и передаёт в `Game`.
- **`Game`** на маунте зовёт `sfxEngine.attach(camera, match.root)`, на анмаунте — `detach()`;
  передаёт движок в `Match` (как `musicEngine`).
- **UI-компоненты** (кнопки, тоглы, лобби) зовут `useSfx().play2D('ui_click'|'ui_hover'|...)`.
- **`Match`** инжектит `ISfxEngine` (опционален: без него — тишина, как с музыкой в юнитах) и дёргает
  его в нужных местах (см. ниже).

## Сетевая модель (кто детектит и играет)

Принцип: **каждый пир озвучивает игрока из доступного ему локально источника; для не-симулируемого
локально игрока — из принятых данных.**

- **Боёвка** (`fired/block/kill/respawn`) — уже эмитится хостом и применяется клиентом. Цепляем SFX к
  этому же пути на ОБОИХ концах → звуки боёвки обоих игроков слышны позиционно. (Свой выстрел у клиента
  — по эху хоста, синхронно с визуалом луча, который и так host-авторитетный.)
- **Свои движения** (прыжок/рывок/щит/land/cooldown) — детект из ЛОКАЛЬНОЙ симуляции своего игрока на
  каждом пире → сразу, без сетевой задержки, позиционно у себя.
- **Движения соперника:**
  - **щит и рывок** — из флагов снапшота (`shielding`, `dashing` уже передаются) по их переходам →
    новых сообщений не нужно;
  - **прыжок и приземление** — новое дискретное событие протокола `{ t:'move'; id; kind:'jump'|'land';
    pos }`, которое хост шлёт для любого игрока; клиент играет его для соперника. Свои move-события
    (id == localId) клиент игнорит (уже сыграл локально). Хост детектит и играет обоих из своей
    симуляции (он симулирует обоих) и шлёт события клиенту.

Детекторы переходов (грунт↔воздух для jump/land, флаги щита/рывка) — в `Match.update`, по игрокам.
Состояние предыдущего кадра хранится per-player (маленькая структура «прошлый grounded/shielding/dashing»).

## Отсчёт и cooldown_ready

- `count_tick` — в `Match` на фазе `countdown`: когда целая секунда обратного отсчёта меняется (3→2→1),
  один `play2D('count_tick')`.
- `cooldown_ready` — локальный игрок: когда кулдаун рывка ЛИБО щита переходит из «не готов» в «готов»,
  один `play2D('cooldown_ready')` (про свой ресурс → 2D). Детект по состоянию кулдаунов своего `Player`.

## Громкости и константы (стартовые, тюнятся)

```
SFX_MASTER_GAIN = 0.9
SFX_GAIN: per-event — геймплей ~0.8–1.0, UI заметно тише (~0.3–0.45), лупы ~0.5
PANNER: refDistance ~6, rolloffFactor ~1, maxDistance ~60 (под арену)
```

## Тестирование

- **Юниты** (`FakeSfxEngine`): логика триггеров — какое событие/переход → какой вызов движка
  (`playAt`/`play2D`/`startLoop`/`stopLoop`); подавление эха своего id у клиента; старт/стоп `shield_loop`
  по переходам; `count_tick` раз на секунду; `cooldown_ready` по готовности любого из двух кулдаунов.
  Конструируем `Match` напрямую с `FakeSfxEngine`, как в `Match.music.test`.
- **e2e-смоук** (`tests/sfx.spec.ts`): движок грузится, `attach` к камере не падает, при входе в матч
  и стрельбе вызовы идут без ошибок (через debug-глобал-счётчик, напр. `__debugSfx` — последние
  сыгранные события — по аналогии с `__debugMusic`).

## Вне области

- `ghost_loop` — нет состояния «ghost form» (не подключаем).
- `menu_loop`, `ui_back`, `beam_loop`, `go`, `match_end`, `victory`/`defeat`/`draw` — редактор их
  намеренно не делал (UI передаёт это иначе); не ждём.
- Музыка в меню — отдельная будущая задача (см. память `audio-next-steps`).
