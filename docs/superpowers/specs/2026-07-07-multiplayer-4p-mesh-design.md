# Мультиплеер до 4 игроков: децентрализованный меш

Дата: 2026-07-07. Статус: дизайн согласован, ждёт имплементационного плана.

## 1. Цель и видение

Расширить OneShot со строгого 1v1 до матчей на 2–4 участника с единой командной моделью:
каждый участник выбирает команду (в UI команда — это просто цвет, без имён; внутренне —
индекс 0..3), раскладка по командам определяет режим —
2v2, 3v1, FFA (все в разных), а 1v1 остаётся вырожденным случаем (2 участника в разных
командах). Отдельного «режима игры» в коде не существует.

Релизная платформа — Steam-билд (Tauri). Веб-версия остаётся 1v1; архитектура
транспорт-агностична, так что веб получит N игроков «бесплатно», если захотим позже.

## 2. Сводка ключевых решений

| Тема | Решение |
|---|---|
| Топология | Полный меш, БЕЗ хоста-авторитета: «у каждого факта ровно один владелец» |
| Команды | Безымянные, игрок выбирает только ЦВЕТ команды; хост выбирает за ботов; любая раскладка, кроме «все в одной» |
| Слоты | Изначально 2; владелец лобби добавляет/убирает до 4 суммарно |
| Боты | Слот заполняет владелец лобби; бот = обычный игрок, симулируемый процессом владельца |
| Цвета игроков | Личная пара (primary, reserve) всегда неизменна, не зависит от сетевой роли |
| Свой/чужой | Плашка с именем цвета команды над моделью; при ровно 2 участниках плашек нет |
| Попадание | Судит стрелок по своей картинке; смерть/блок судит жертва («щит побеждает») |
| Friendly fire | Луч блокируется телом тиммейта, но не вредит ему |
| Конец матча | Время вышло ЛИБО в матче осталась одна команда |
| Победа | Таблица мест: команды по сумме киллов, внутри — по личным |
| Стрики/SINGULARITY | Без изменений: каждый пир выводит их из канонического потока `died` |
| Читы | Осознанно не защищаемся (игра с друзьями); пересмотр — при публичном матчмейкинге |

## 3. Комната и лобби

- `RoomSession` (`src/net/RoomSession.ts`) перестраивается с «host + один opponent» на
  массив слотов. Слотов изначально **2**; владелец лобби добавляет слоты до 4 суммарно
  (включая себя) и убирает пустые. Дефолтный вид комнаты = сегодняшний 1v1.
- id участника = индекс слота (0..3), создатель лобби всегда в слоте 0 (`HOST_ID = 0`
  сохраняется как «id создателя»). Константа `OPPONENT_ID` упраздняется.
- `RosterEntry` += `team: 0..3` и `reserveColor`. Полный ростер (вкл. число слотов) —
  факт создателя лобби: он сажает входящих в первый свободный слот (`Assign.yourId`
  становится динамическим), валидирует смену команд, управляет ботами.
- Человек заходит: свободный слот есть → занимает его; нет → отказ «room full».
  Никакого автоматического выселения ботов — слоты управляются явно.
- Команда по умолчанию при входе — первая незанятая (комната «из коробки» FFA).
  Человек меняет только свою команду (`setTeam` → создатель → broadcast ростера);
  команды ботов меняет создатель.
- `canStart` = занято ≥ 2 слотов И среди занятых ≥ 2 разные команды.
- Ready-ритуал `ready → countdown → live` не меняется (уже написан через Set по id);
  боты авто-ready. Штамп старта countdown ставит создатель лобби.

## 4. Цвета и индикация свой/чужой

**Личный цвет неприкосновенен.** Пара `(primaryColor, reserveColor)` — атрибут
внешности игрока: primary красит тело/луч/эффекты, reserve сегодня используется
кольцом планеты, будущие модели шара могут использовать его иначе. Пара целиком
едет в `RosterEntry` (сейчас reserve выбрасывается — чинится), каждый клиент рендерит
каждого игрока его настоящей парой. `assignColor` (подмена цвета клиента при коллизии
с хостом, `RoomSession.ts:110-116`) удаляется: цвета не зависят от сетевой роли.
Коллизии цветов моделей допустимы — различение делают плашки. Ботам пара генерится
в `botAppearance`.

**Команда — это цвет плашки, и только плашки.** Отдельная палитра `TEAM_COLORS`
(4 фиксированных цвета), никак не связанная с цветами моделей. Над моделью игрока —
billboard-плашка (THREE.Sprite, canvas-текстура: имя на фоне цвета команды),
`userData.noRaycast`, живёт в группе визуалов игрока, скрывается вместе с моделью.

**Показ плашек:** ровно 2 участника → плашек нет (текущий 1v1 сохраняется
пиксель-в-пиксель); 3–4 → плашки над всеми, кроме своей модели.

В лобби у слота два независимых атрибута: личный цвет модели и чип команды из
`TEAM_COLORS` (клик по чипу — смена команды).

## 5. Сетевая модель: меш и владение фактами

Топология — полный меш (4 пира = 6 линков). На Steam связность решает SDR;
`BroadcastChannelNet` — шина, меш там естественен; `LoopbackNet` учится N пирам.
В Rust поднимается `LOBBY_MAX_MEMBERS` 2→4 (`src-tauri/src/steam_net.rs:24`).

Принцип: **у каждого факта ровно один владелец**, консенсус не нужен:

| Факт | Владелец |
|---|---|
| Движение, снапшоты игрока | Сам игрок (KCC у себя — как сегодня у клиента) |
| Смерть, блок, респавн игрока | Сам игрок («никто не объявит меня мёртвым, кроме меня») |
| Попадание луча | Стрелок (судит по своей картинке) |
| Бот целиком (интенты, снапшоты, его смерти) | Процесс владельца лобби |
| Ростер, слоты, команды, штамп старта | Создатель лобби |
| Таймер матча | Никто: конец детерминирован от штампа старта |
| Счёт, стрики, SINGULARITY | Никто: каждый выводит из канонического потока `died` |

Следствия:

- Лаг-комп история с перемоткой (`history`, `viewTick`) **удаляется**: стрелок
  стреляет по интерполированным ремоутам — по тому, что видит.
- Общий клок не нужен: у каждой пары пиров свой интерполяционный буфер
  (существующий pairwise clock-nudge обобщается до Map по пиру).
- Снапшоты: каждый пир рассылает состояние своих игроков (себя + своих ботов)
  всем; формат `PlayerSnapshot` переиспользуется.
- Роли `host|client` в `Match` заменяются одной симметричной ролью `peer`.
- Уход любого пира — не событие для правил: его сущности замирают и удаляются,
  счёт остаётся в таблице, матч живёт, пока представлены ≥ 2 команды.
  Класс проблем «хост завис/свернулся — все замерли» исчезает: свернувший
  замирает сам. Боты уходят вместе со своим владельцем: пир, симулировавший
  ботов, ливнул → его боты пропадают из матча тем же ходом (переназначение
  ботов — v2). Уход владельца лобби в матче: таймер дотикает детерминированно,
  матч продолжается; в лобби — комната закрывается (миграция владельца
  лобби — out of scope v1).

## 6. Протокол выстрела и смерти

1. Стрелок на своём тике делает raycast в своём мире против интерполированных
   ремоутов. Raycast видит ВСЕХ (тиммейт телом блокирует линию огня — тактика).
2. Луч упёрся в тиммейта → выстрел гаснет на нём: ни урона, ни claim'а. Промах →
   только `fired` (визуал луча у всех). Попадание во врага X → `fired` + claim «hit X».
3. **X судит сам** (владелец своей смерти): жив? не призрак? щит?
   - Щит активен на экране X к моменту прихода claim'а → `blocked {shooter}`.
     Правило «щит побеждает»: защитник, успевший нажать щит на своём экране,
     никогда не умирает. История щита и клок-перемотка не нужны.
   - Иначе → `died {victim, by}` + X применяет к себе нокбэк/смерть.
   - Claim от однокомандника (рассинхрон составов) → игнорируется.
4. Все пиры применяют `died`/`blocked` одинаково: труп/гост, +1 стрелку, стрики,
   OVERHEAT (по собственной серии игрока), SINGULARITY (у кого-то стрик ≥ 5 —
   прострел стен + прозрачные блоки у всех) — механика `applyComeback` не меняется.
5. Двойной килл (два claim'а в одном окне): X мёртв после первого, второй гаснет.
   Подтверждение килла у стрелка — через RTT (клиенты и сегодня так живут).

Транспортные требования: каналы надёжные (Trystero/Steam — да), имена экшенов
≤ 12 байт. Новые/изменённые сообщения: `RosterEntry{+team,+reserveColor}`,
динамический `Assign.yourId`, `setTeam`, число слотов в ростере, claim попадания,
`died`, `blocked`, `left` — детализация форматов в имплементационном плане.

## 7. Правила симуляции

- **Счёт**: киллы/смерти — из потока `died`; командный счёт = сумма киллов состава.
  Итог: команды по сумме, внутри — по личным киллам. Локальный исход: твоя команда
  единолично первая → win; делит первое → draw; иначе → lose (существующие экраны
  переиспользуются). `computeResult` (`Match.ts:846`) переписывается с бинарного
  сравнения на ранжирование.
- **Уход игрока в матче**: его шар пропадает, строка в таблице остаётся
  (со счётом) с пометкой «покинул»; его боты уходят вместе с ним.
- **Конец матча**: время вышло ЛИБО в матче осталась только одна команда
  (немедленный конец с текущей таблицей).
- **Респавн**: механика не меняется (призрак → возрождение на месте остановки),
  исполняет владелец игрока, остальные видят через снапшоты/событие.
- **Боты**: `BotController` получает выбор цели — ближайший живой не-тиммейт
  (сейчас целится в единственного оппонента).
- **Нокбэк**: жертва применяет к себе сама при приёме claim'а.

## 8. Спавны и карты

Тип карты: `spawns: [Vec3, Vec3]` → `Vec3[]` длиной 4 (валидация редактора
«ровно 2» → «ровно 4», `editorStore.ts:198`); в 4 существующие карты добавляется
по 2 точки. Стартовое размещение: слот i → точка i. Размер арены не трогаем до
плейтеста.

**Открытый вопрос (обсудить отдельно до плейтеста `feature/lobby-slots`):**
расстановка спавн-точек на существующих картах и правила выбора точки
(командная группировка? удалённость от врагов?).

## 9. HUD и лобби — контракт данных

Визуальный дизайн таблицы счёта, лобби и киллфида — отдельные сессии через
frontend-design (НЕ в этой спеке). Здесь — только контракт данных:

- Счёт в `useGameHUD`: ключ по **id** (сейчас по имени — коллизии), поля
  `team`, `left`. Группировка и командные суммы вычислимы.
- Киллфид: новый поток «кто кого» (killer, victim, blocked) из `died`/`blocked`.
- Ready-оверлей: список участников с ready-статусами вместо жёстких двух углов.
- Экран конца: ранжированная таблица + локальный win/draw/lose.
- Плашки (секция 4): имя + цвет команды, при > 2 участниках.
- Лобби: до 4 слотов (имя, личный цвет, чип команды, ready; у создателя — кнопки
  слотов и ботов), `canStart` из секции 3.

## 10. Тестовая стратегия

- **Юниты (основная нагрузка)** — `LoopbackNet` на 3–4 пира в одном процессе:
  протокол смерти (двойной килл, «щит побеждает», claim тиммейта игнорируется),
  сходимость счёта у всех пиров, уход пира, боты владельца, `RoomSession`
  (слоты/команды/canStart), ранжирование `computeResult`.
- **E2e** — комната на 3 вкладки через `BroadcastChannelNet`: вход, команды,
  старт, плашки, выход посреди матча. Боевых сценариев в e2e — минимум
  (двухвкладочные боевые спеки уже флакают; вся боёвка — в юнитах).
- ~20 существующих тест-файлов ссылаются на HOST_ID/OPPONENT_ID/двухспавновые
  карты — правятся по ходу веток.

## 11. Ветки и порядок работ

Каждая ветка независимо играбельна и мержится в release по готовности:

1. **`feature/colors-rework`** — пара цветов в ростер, удаление `assignColor`,
   reserve у ботов. Маленькая, независимая.
2. **`feature/lobby-slots`** — слоты 2→4, команды, `TEAM_COLORS`, плашки,
   лобби-UI, `canStart`, спавны `Vec3[]`. На текущей звезде уже даёт играбельное
   «я + до 3 ботов» (хост умеет симулировать ботов).
3. **`feature/mesh-net`** — децентрализация: симметричный `Match`, протокол
   смерти, per-peer интерполяция, удаление лаг-комп истории, `left`-семантика.
   Самая большая.
4. **`feature/steam-lobby4`** — Rust cap 2→4 + Steam-лобби UI (стадия 2
   Steam-интеграции).

## 12. Осознанные компромиссы / out of scope

- Читы тривиальны (пир объявляет что хочет) — принято для игры с друзьями;
  пересмотр при публичном матчмейкинге.
- Окна рассогласования в доли секунды (жертва уже видит блок, стрелок ещё
  видит попадание) — цена децентрализации, гасится юнит-покрытием протокола.
- Demo-запись становится «точкой зрения пира», а не абсолютной истиной.
- Steam quick match остаётся парным (1v1); N игроков — только через лобби/инвайты.
- Миграция владельца лобби, переназначение ботов при уходе владельца — v2.
- Веб-версия остаётся 1v1 (транспортно совместима, но в релиз не входит).
