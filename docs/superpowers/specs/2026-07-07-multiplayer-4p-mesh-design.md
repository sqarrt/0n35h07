# Мультиплеер до 4 игроков: децентрализованный меш

Дата: 2026-07-07 (ревизия 2 — режимы вместо свободного выбора команд). Статус: дизайн согласован.

## 1. Цель и видение

Расширить OneShot со строгого 1v1 до матчей на 2–4 участника. Хост выбирает в лобби
один из трёх **режимов**: `1v1`, `2v2`, `FFA` (все против всех). Ключевой принцип:
режим — это **пресет лобби** (число слотов, раскладка команд, правило спавна, canStart);
**внутри симуляции режимов не существует** — она всегда командная:

| Режим | Внутреннее представление | Команды в UI |
|---|---|---|
| 1v1 | 2 команды по 1 игроку | не видны |
| 2v2 | 2 команды по 2 игрока | видны (2 цвета) |
| FFA (2–4) | N команд по 1 игроку | не видны |

Боевые правила, счёт, конец матча — один универсальный код без ветвлений по режиму.

Релизная платформа — Steam-билд (Tauri). Веб-версия остаётся 1v1; архитектура
транспорт-агностична, так что веб получит N игроков «бесплатно», если захотим позже.

## 2. Сводка ключевых решений

| Тема | Решение |
|---|---|
| Топология | Полный меш, БЕЗ хоста-авторитета: «у каждого факта ровно один владелец» |
| Режим | Выбирает хост в лобби: 1v1 / 2v2 / FFA; внутри симуляции — всегда команды |
| Слоты | 1v1 → 2; 2v2 → 4 (две пары-команды); FFA → 4 (старт от 2 занятых) |
| Рассадка 2v2 | Слоты сгруппированы в команды; клик по свободному слоту = пересесть; ботов сажает хост |
| Боты | Слот заполняет хост; бот = обычный игрок, симулируемый процессом владельца |
| Цвета игроков | Личная пара (primary, reserve) всегда неизменна, не зависит от сетевой роли — СДЕЛАНО (colors-rework) |
| Свой/чужой | Плашка с именем над моделью: 1v1 — нет; 2v2 — цвет команды; FFA — нейтральная |
| Попадание | Судит стрелок по своей картинке; смерть/блок судит жертва («щит побеждает») |
| Friendly fire | Луч блокируется телом тиммейта, но не вредит ему (актуально только в 2v2) |
| Спавны | Карты остаются с 2 точками; 1v1 — как сейчас; 2v2 — команды кучками у точек 0/1; FFA — рандом с мин-дистанцией |
| Конец матча | Время вышло ЛИБО в матче осталась одна команда |
| Победа | Таблица мест: команды по сумме киллов, внутри — по личным (в 1v1/FFA вырождается в личную) |
| Стрики/SINGULARITY | Без изменений: каждый пир выводит их из канонического потока `died` |
| Читы | Осознанно не защищаемся (игра с друзьями); пересмотр — при публичном матчмейкинге |

## 3. Комната и лобби

- Хост выбирает **режим** (1v1 / 2v2 / FFA); режим едет в ростер-broadcast. Смена
  режима заблокирована, пока участников больше, чем слотов нового режима (никого
  не выселяем молча).
- `RoomSession` перестраивается с «host + один opponent» на массив слотов:
  - **1v1**: 2 слота — ровно текущая комната.
  - **2v2**: 4 слота, сгруппированы в 2 команды (слоты 0-1 = команда 0, слоты 2-3 =
    команда 1). Хост всегда в слоте 0.
  - **FFA**: 4 слота без групп, каждый занятый слот — своя команда.
- id участника = индекс слота, создатель лобби в слоте 0 (`HOST_ID = 0` сохраняется).
  Константа `OPPONENT_ID` упраздняется.
- Зашедший человек занимает первый свободный слот (`Assign.yourId` — динамический).
  Свободных нет → отказ «room full». Никакого выселения ботов — слоты явные.
- **Пересадка**: клик по свободному слоту → сообщение `setSlot { slot }` → хост
  валидирует (слот свободен) → broadcast ростера. Хост пересаживает ботов сам.
  Осмысленно в 2v2 (смена команды); в FFA безвредно.
- `canStart`: 1v1 — оба слота заняты; 2v2 — все 4 заняты; FFA — занято ≥ 2.
- Ready-ритуал `ready → countdown → live` не меняется; боты авто-ready. Штамп
  старта countdown ставит создатель лобби; для FFA в Start-сообщение он же кладёт
  сгенерированные спавн-позиции (секция 8).

## 4. Цвета и индикация свой/чужой

**Личный цвет неприкосновенен** — реализовано в `feature/colors-rework` (смержено):
пара `(primaryColor, reserveColor)` едет в `RosterEntry` целиком, подмена при
коллизии удалена, у ботов детерминированный reserve. Коллизии цветов моделей
допустимы — различение делают плашки.

**Плашки** — billboard над моделью (THREE.Sprite, canvas-текстура: имя),
`userData.noRaycast`, в группе визуалов игрока, скрывается вместе с моделью:

- **1v1** — плашек нет (текущий вид игры сохраняется пиксель-в-пиксель).
- **2v2** — плашка цвета команды; `TEAM_COLORS` — 2 фиксированных цвета, никак
  не связанных с цветами моделей. Над своей моделью плашки нет.
- **FFA** — нейтральная плашка (белая/серая) с именем: все враги, цветовое
  кодирование не нужно, имя — для атрибуции киллов. Над своей — нет.

В лобби 2v2 группы слотов подкрашены `TEAM_COLORS`; личный цвет модели игрока —
отдельный атрибут, виден в слоте.

## 5. Сетевая модель: меш и владение фактами

Топология — полный меш (4 пира = 6 линков). На Steam связность решает SDR;
`BroadcastChannelNet` — шина, меш там естественен; `LoopbackNet` учится N пирам.
В Rust поднимается `LOBBY_MAX_MEMBERS` 2→4 (`src-tauri/src/steam_net.rs:24`).

Принцип: **у каждого факта ровно один владелец**, консенсус не нужен:

| Факт | Владелец |
|---|---|
| Движение, снапшоты игрока | Сам игрок (KCC у себя — как сегодня у клиента) |
| Смерть, блок, респавн игрока | Сам игрок («никто не объявит меня мёртвым, кроме меня») |
| Попадание луча | Стрелок (судит по своей картинке) |
| Бот целиком (интенты, снапшоты, его смерти) | Процесс владельца лобби |
| Ростер, слоты, режим, штамп старта, FFA-спавны | Создатель лобби |
| Таймер матча | Никто: конец детерминирован от штампа старта |
| Счёт, стрики, SINGULARITY | Никто: каждый выводит из канонического потока `died` |

Следствия:

- Лаг-комп история с перемоткой (`history`, `viewTick`) **удаляется**: стрелок
  стреляет по интерполированным ремоутам — по тому, что видит.
- Общий клок не нужен: у каждой пары пиров свой интерполяционный буфер
  (существующий pairwise clock-nudge обобщается до Map по пиру).
- Снапшоты: каждый пир рассылает состояние своих игроков (себя + своих ботов) всем.
- Роли `host|client` в `Match` заменяются одной симметричной ролью `peer`.
- Уход любого пира — не событие для правил: его сущности замирают и удаляются,
  счёт остаётся в таблице, матч живёт, пока представлены ≥ 2 команды.
  Боты уходят вместе со своим владельцем (переназначение — v2). Уход владельца
  лобби в матче: таймер дотикает детерминированно, матч продолжается; в лобби —
  комната закрывается (миграция владельца лобби — out of scope v1).

## 6. Протокол выстрела и смерти

1. Стрелок на своём тике делает raycast в своём мире против интерполированных
   ремоутов. Raycast видит ВСЕХ (тиммейт телом блокирует линию огня — тактика).
2. Луч упёрся в тиммейта → выстрел гаснет на нём: ни урона, ни claim'а. Промах →
   только `fired` (визуал луча у всех). Попадание во врага X → `fired` + claim «hit X».
3. **X судит сам** (владелец своей смерти): жив? не призрак? щит?
   - Щит активен на экране X к моменту прихода claim'а → `blocked {shooter}`.
     Правило «щит побеждает»: защитник, успевший нажать щит на своём экране,
     никогда не умирает. История щита и клок-перемотка не нужны.
   - Иначе → `died {victim, by}` + X применяет к себе нокбэк/смерть.
   - Claim от однокомандника (рассинхрон составов) → игнорируется.
4. Все пиры применяют `died`/`blocked` одинаково: труп/гост, +1 стрелку, стрики,
   OVERHEAT (по собственной серии игрока), SINGULARITY (у кого-то стрик ≥ 5 —
   прострел стен + прозрачные блоки у всех) — механика `applyComeback` не меняется.
5. Двойной килл (два claim'а в одном окне): X мёртв после первого, второй гаснет.
   Подтверждение килла у стрелка — через RTT (клиенты и сегодня так живут).

Транспортные требования: каналы надёжные (Trystero/Steam — да), имена экшенов
≤ 12 байт. Новые/изменённые сообщения: `RosterEntry{+reserveColor}` (готово),
режим в ростер-broadcast, динамический `Assign.yourId`, `setSlot`, claim
попадания, `died`, `blocked`, `left`, FFA-спавны в `Start` — детализация
форматов в имплементационных планах.

## 7. Правила симуляции

- **Команда** = группа слотов из пресета режима; симуляция ветвлений по режиму
  не имеет. Тиммейт-логика (блокировка луча телом, игнор claim'а) в 1v1/FFA
  просто никогда не срабатывает (команды по одному).
- **Счёт**: киллы/смерти — из потока `died`; командный счёт = сумма киллов состава.
  Итог: команды по сумме, внутри — по личным киллам. В 1v1/FFA «команда» = игрок —
  таблица вырождается в личную. Локальный исход: твоя команда единолично первая →
  win; делит первое → draw; иначе → lose. `computeResult` переписывается с
  бинарного сравнения на ранжирование.
- **Уход игрока в матче**: его шар пропадает, строка в таблице остаётся (со
  счётом) с пометкой «покинул»; его боты уходят вместе с ним.
- **Конец матча**: время вышло ЛИБО осталась только одна команда (немедленный
  конец с текущей таблицей).
- **Респавн**: механика не меняется (призрак → возрождение на месте остановки),
  исполняет владелец игрока.
- **Боты**: `BotController` получает выбор цели — ближайший живой не-тиммейт.
- **Нокбэк**: жертва применяет к себе сама при приёме claim'а.

## 8. Спавны

**Карты не меняются**: тип `spawns: [Vec3, Vec3]`, редактор и все 4 карты
остаются как есть. Правило стартового размещения задаёт режим:

- **1v1** — слот 0 → точка 0, слот 1 → точка 1 (ровно как сейчас).
- **2v2** — команда 0 кучкой у точки 0, команда 1 кучкой у точки 1; внутри кучки
  небольшие фиксированные офсеты (именованная константа), чтобы капсулы не
  слипались в физике.
- **FFA** — случайные позиции в пределах арены с минимальной попарной дистанцией
  (именованная константа). Генерирует **создатель лобби** при старте и кладёт в
  `Start`-сообщение — у всех пиров идентичные позиции без общего RNG.

Размер арены не трогаем до плейтеста.

## 9. HUD и лобби — контракт данных

Визуальный дизайн таблицы счёта, лобби и киллфида — отдельные сессии через
frontend-design (НЕ в этой спеке). Контракт данных:

- Счёт в `useGameHUD`: ключ по **id**, поля `team`, `left`. Группировка по
  командам (2v2) или плоский список (1v1/FFA) — вычислимо из режима и ростера.
- Киллфид: поток «кто кого» (killer, victim, blocked) из `died`/`blocked`.
- Ready-оверлей: список участников с ready-статусами вместо жёстких двух углов.
- Экран конца: ранжированная таблица + локальный win/draw/lose.
- Плашки (секция 4): 2v2 — цвет команды, FFA — нейтральные, 1v1 — нет.
- Лобби: выбор режима (хост), слоты по пресету (имя, личный цвет, ready; в 2v2 —
  группировка по командам с подкраской `TEAM_COLORS`; кнопки хоста для ботов;
  клик по свободному слоту — пересадка), `canStart` из секции 3.

## 10. Тестовая стратегия

- **Юниты (основная нагрузка)** — `LoopbackNet` на 3–4 пира в одном процессе:
  протокол смерти (двойной килл, «щит побеждает», claim тиммейта игнорируется),
  сходимость счёта у всех пиров, уход пира, боты владельца, `RoomSession`
  (режимы/слоты/пересадка/canStart), ранжирование `computeResult`, спавн-правила
  (2v2 кучки, FFA мин-дистанция).
- **E2e** — комната на 3 вкладки через `BroadcastChannelNet`: вход, режимы,
  пересадка, старт 2v2/FFA, плашки, выход посреди матча. Боевых сценариев в e2e —
  минимум (двухвкладочные боевые спеки уже флакают; вся боёвка — в юнитах).
- ~20 существующих тест-файлов ссылаются на HOST_ID/OPPONENT_ID — правятся по
  ходу веток; тесты карт (`spawns.length===2`) теперь НЕ меняются.

## 11. Ветки и порядок работ

1. **`feature/colors-rework`** — ГОТОВА, смержена в release_1.1.0 (ef984bb).
2. **`feature/lobby-slots`** — режимы, слоты, пересадка, `TEAM_COLORS`, плашки,
   спавн-правила, лобби-UI, `canStart`. На текущей звезде уже даёт играбельное
   «я + до 3 ботов» (хост умеет симулировать ботов). Карты/редактор — вне scope.
3. **`feature/mesh-net`** — децентрализация: симметричный `Match`, протокол
   смерти, per-peer интерполяция, удаление лаг-комп истории, `left`-семантика.
4. **`feature/steam-lobby4`** — Rust cap 2→4 + Steam-лобби UI (стадия 2
   Steam-интеграции).

## 12. Осознанные компромиссы / out of scope

- Читы тривиальны (пир объявляет что хочет) — принято для игры с друзьями.
- Окна рассогласования в доли секунды — цена децентрализации, гасится юнитами.
- Demo-запись становится «точкой зрения пира», а не абсолютной истиной.
- Steam quick match остаётся парным (1v1); N игроков — только через лобби/инвайты.
- Миграция владельца лобби, переназначение ботов при уходе владельца — v2.
- Веб-версия остаётся 1v1 (транспортно совместима, но в релиз не входит).
