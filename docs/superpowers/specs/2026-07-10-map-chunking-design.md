# Чанки геометрии карты + frustum culling

Дата: 2026-07-10 · Ветка: `feature/map-perf-chunks` (от `release_1.1.0`)

Под-проект №1 из трёх по оптимизации больших карт (далее: скрытые грани, статичные тени).

## Проблема

Большая карта (os_test ~9500 блоков / ~300k вершин) рендерится как 3–4 слитых меша на всю
арену. Слитый меш нельзя отсечь по частям — он целиком уходит в каждый проход (основной, тени,
обводка), даже если в кадре видна четверть карты. На 300k×(3 прохода) это ~15 fps против 120 на
мелких картах. Цель — дать GPU отсекать невидимую часть карты во всех проходах.

## Решение

Разбить **визуальную** геометрию карты на пространственные чанки — отдельные меши, которые
three.js отсекает по frustum сам. Коллайдер остаётся одним слитым trimesh (физике culling не
нужен, у Rapier свой BVH).

### Чанкинг

- Сетка по X/Z, чанк — квадрат `CHUNK_SIZE = 8` мировых единиц (16 ячеек) на всю высоту Y.
- Блок относится к чанку по **центру** `pos` (пересекающие границу не режем — AABB чанка чуть
  больше, на корректность culling не влияет).
- Каждый чанк несёт свои 4 группы `(opaque|transparent)×(raycast|noRaycast)`. Пустые группы/чанки
  не создаются.
- Коллайдер — одно поле на карту (слитый trimesh всех непроходимых блоков, как сейчас).

### Формат `CompiledMap` (mapGeometryCache.ts)

```ts
export interface ChunkGeo {
  opaqueRaycast: GeoArrays | null
  opaqueNoRaycast: GeoArrays | null
  transparentRaycast: GeoArrays | null
  transparentNoRaycast: GeoArrays | null
}
export interface CompiledMap {
  chunks: ChunkGeo[]
  collider: GeoArrays | null
}
```

`isEmptyCompiled` = нет чанков (или все пустые) и нет коллайдера. `serializeGeo`/`parseGeo`
сериализуют `{ chunks: SerChunk[], collider }`. **`parseGeo` защитный**: если у объекта нет поля
`chunks` (старый формат) — возвращает пустую `CompiledMap` → рантайм-фолбэк перекомпилирует из
`blocks`. Это не версионирование, а существующее поведение (см. комментарий у `isEmptyCompiled`
про «outdated geo.json format»).

### Компиляция (blockGeometry.ts)

`bucketedBlockGeometries` разбивает блоки по чанкам (по центру) и для каждого чанка мёржит 4
группы; коллайдер собирается из всех непроходимых блоков в один. Возвращает `{ chunks, collider }`.
Клинья/прозрачные/passable/on-side едут через ту же геометрию — чанкинг их не различает, форма не
меняется.

### Рендер (Arena.tsx)

Вместо 4 мешей — цикл по `compiled.chunks`, у каждого до 4 мешей. Raycast-группы (цели луча) —
как сейчас, с per-chunk BVH (`computeBoundsTree`); `World.raycast` уже обходит все меши сцены.
Коллайдер — один trimesh, как сейчас. Пол/сетка/невидимые стены не меняются.

### Пересборка geo.json заранее

Все карты пересобираются в новый формат **до игры**, не в рантайме. Механизм — dev-эндпоинт
`GET /__recompile-maps` в плагине `editorMaps` (`build/vite-plugins/editorMaps.ts`): на живом
dev-сервере через `server.ssrLoadModule('/src/game/mapGeometryCache.ts')` грузит настоящий
пайплайн, перекомпилирует `raw.json` → `geo.json` каждой карты в новом формате и пишет их. Так
переиспользуется реальный код компиляции (DRY), без реимплементации. Эндпоинт запускается один раз
(headless dev-сервер + curl), результат — новые `geo.json` — коммитится.

Рантайм-фолбэк `compileBlocksCached` (компиляция из `blocks` при отсутствии/устаревании geo)
использует тот же новый пайплайн — карты без свежего geo.json не ломаются, просто компилируются в
памяти при монтировании.

## Тесты

- Юнит `tests/unit/mapChunks.test.ts` (или в существующем): блок относится к чанку по центру;
  объединение вершин всех чанков = вершины единой компиляции (нет потери/дублей); пустые чанки
  отсутствуют; коллайдер один и покрывает все непроходимые блоки.
- geo round-trip нового формата: `parseGeo(serializeGeo(c))` эквивалентно `c`; `parseGeo` старого
  формата (без `chunks`) → пустая `CompiledMap`.
- Игровой fps на os_test — руками (пользователь).

## Вне объёма (отдельные под-проекты)

- Скрытые грани между сплошными ячейками — следующий под-проект.
- Статичные тени — последний под-проект (переоценить после замера прироста от чанков).
- Чанкинг коллайдера — не нужно.

## Процесс

- Ветка `feature/map-perf-chunks`; тесты запускает пользователь; агент — `tsc`, `lint` и
  пересборку geo.json через headless dev-сервер. Файлы `src/maps/os_test/*` в рабочем дереве —
  живые правки пользователя, не трогаю (пересобираю geo только committed-картам через эндпоинт).
